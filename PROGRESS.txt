## Session 71 - 2026-04-11

### Travail effectue

**Revue securite acces paywall + ajustement tarifaire**

#### Audit securite acces control

Objectif : s'assurer qu'un appel direct aux Server Actions ou aux routes API ne peut jamais debloquer gratuitement une fonctionnalite STARTER / PRO / BUSINESS.

**Vulnerabilite critique corrigee — `src/actions/organizations.ts`**

Les 5 mutations de gestion d'equipe (`addUserToOrganization`, `updateUserRole`, `updateUserModulePermissions`, `updateUserNotificationPreference`, `removeUserFromOrganization`) verifiaient l'appartenance a l'org et le role OWNER mais pas l'entitlement `TEAM_ROLES` (BUSINESS uniquement). Un OWNER sur un plan inferieur pouvait appeler ces actions directement (DevTools / curl) et gerer des membres sans souscrire.

Correctif : ajout du gate `resolveEntitlementGate(subscription, "TEAM_ROLES")` dans chacune des 5 fonctions, apres `requireOrganizationModuleContext`.

**Vulnerabilite haute corrigee — `src/actions/predictive.ts` (fonctions Internal)**

`getStockPredictionsInternal`, `getBatchMortalityPredictionsInternal`, `getBatchMarginPredictionsInternal` etaient exportees depuis un fichier `"use server"`. En Next.js 14, tout export d'un tel fichier est une Server Action callable via POST sans les gates d'abonnement et de membership. N'importe quel utilisateur authentifie pouvait appeler ces fonctions avec l'`organizationId` d'une org tierce.

Correctif : extraction des 3 fonctions de calcul dans `src/lib/predictive-compute.ts` (sans `"use server"`). Suppression des exports Internal de `predictive.ts`. `notifications.ts` et `business.ts` importent desormais depuis le nouveau module lib.

**Vulnerabilite moyenne corrigee — `getStockTrends`**

Verifiait l'acces module STOCK mais pas l'entitlement `PREDICTIVE_STOCK_ALERTS` (PRO+). La page stock gatait correctement au niveau UI, mais un appel direct a la Server Action contournait ce controle.

Correctif : ajout du gate `PREDICTIVE_STOCK_ALERTS` dans `getStockTrends`, identique au pattern de `getStockPredictions`.

**Points confirmes sains lors de l'audit**

- `getOrganizationSubscription` : toujours DB-fetched, jamais de confiance client
- `gateHasFullAccess` : retourne `true` uniquement si `access === "full"`, "preview" est bloque
- `profitability.ts` : gate avant tout calcul
- Les 3 routes API `/api/reports/*` : auth + membership + module + subscription gate a chaque niveau
- Server Components : donnees financieres jamais serialisees vers le client

#### Ajustement tarifaire

- **PRO** : 8 000 → 10 000 FCFA / mois (modifie dans `offer-catalog.ts` et `README.md`)
- **BUSINESS** : 20 000 → 25 000 FCFA / mois (modifie dans `offer-catalog.ts` et `README.md`)
- `subscriptions.ts` avait deja les nouveaux prix, seul `offer-catalog.ts` etait decale.

#### Correctif build Vercel

Turbopack traitait `export type { StockPredictionsResult }` dans `predictive.ts` comme une valeur d'action exportee et echouait a en creer un proxy runtime. Suppression de cette re-export de type — les types ne doivent pas etre exportes depuis des fichiers `"use server"`.

**Regle retenue** : dans un fichier `"use server"`, n'exporter que des fonctions async. Jamais de types, interfaces ou valeurs non-action.

### Resultat

- Aucune fonctionnalite STARTER / PRO / BUSINESS n'est deblocable par appel direct aux Server Actions
- `tsc --noEmit` : OK
- `next build` : OK (Turbopack)
- Prix PRO et BUSINESS mis a jour dans le catalogue et le README

### Prochaine session recommandee

- Appliquer la meme discipline de gate dans tout futur Server Action manipulant des donnees premium
- Surveiller les exports des fichiers `"use server"` : uniquement des fonctions async

- **`src/actions/analytics.ts`** : Server Action `trackAlertAction` pour les evenements client-side depuis `NotificationDropdown`.
- **`app/api/track/pricing-cta/route.ts`** : route redirect — enregistre `pricing_cta_clicked` cote serveur puis redirige vers WhatsApp, sans JavaScript client.

- **5 paywalls instrumentes** : `batches/[id]/page.tsx` (profitabilite, mortalite, marge), `reports/page.tsx`, `business/page.tsx`, `batches/new/page.tsx`, `team/page.tsx`.
- **3 routes export instrumentees** : batch PDF, mensuel PDF/Excel/CSV, Business Excel/CSV.
- **Page `/pricing`** : tracking `pricing_page_visited` avec contexte `from=`.
- **`NotificationDropdown`** : `handleNavigate` appelle `trackAlertAction` avant `router.push`.

#### Sous-lot 2 : Exploitation funnel

- **Evenement `subscription_activated`** ajoute au type `AnalyticsEventName`.
- **`src/lib/subscription-lifecycle.ts`** : `activateOrganizationSubscription` accepte maintenant `userId?` et `triggeredBy?`, appelle `track()` apres l'upsert. Couvre les 3 chemins : `user_confirm`, `admin_direct`, `admin_wave`.
- **`docs/analytics/funnel-queries.sql`** : 7 requetes pret-a-l'emploi — funnel brut, conversion par surface, conversion par plan cible, drop-off par etape, breakdown activations, cohorte hebdomadaire, audit de coherence.

3. **Boite noire WhatsApp** — gap entre clic CTA et activation non mesurable :
   - Ajout evenement `subscription_payment_requested` dans `createSubscriptionPaymentRequest`
   - Proprietes : `{ plan, amountFcfa, paymentMethod }`
   - Requetes SQL mises a jour pour inclure cette 4e etape du funnel

**README.md mis a jour** : section Pricing corrigee (FREE/STARTER/PRO/BUSINESS avec vrais prix), section Instrumentation produit et funnel ajoutee.

### Resultat

## Session 68 - 2026-04-08

### Travail effectue

**Intelligence Collective — Phase A** : Mise en place des fondations du systeme d'intelligence collective qui transforme SunuFarm d'un simple agregateur de donnees en un systeme qui apprend de l'ensemble des exploitations pour ameliorer ses recommendations.

- **3 nouveaux modeles Prisma** :
  - `BatchOutcomeSnapshot` — snapshot anonymise d'un lot ferme (PAS d'organizationId, ni de batchId), contient mortalite finale, FCR, marge, profil meteo, signaux sanitaires
  - `LearnedPattern` — patterns detectes depuis le pool avec confidence adaptive selon les feedbacks terrain
  - `RecommendationFeedback` — boucle de feedback : chaque conseil suivi ou ignore permet d'ajuster la confiance du pattern
- **Migration SQL creee et appliquee** : `20260408190000_add_collective_intelligence_models`
- **`src/lib/collective-intelligence.ts`** — engine de generation des snapshots :
  - collecte 10 sources de donnees en parallele (mortalite, aliment, meteo, traitements, vaccins, ventes, depenses, poids)
  - calcule FCR, taux de mortalite, marge, stress thermique, retards vaccinaux, et derive de region depuis l'adresse de la ferme
  - expose `generateBatchOutcomeSnapshot()` (never-throws, fire-and-forget) et `backfillBatchOutcomeSnapshots()`
- **`src/lib/collective-benchmark.ts`** — benchmark engine avec strategie de fallback progressive :
  - Niveau 1 : précis (meme race + region + saison), minimum 10 snapshots
  - Niveau 2 : elargi (meme race), minimum 5 snapshots
  - Niveau 3 : global (meme type de lot)
  - null si le pool est trop petit — gere cote appelant
  - `formatCollectiveBenchmarkInsight()` : formate un message d'insight compare pour l'UI
- **Hook `closeBatch`** : un appel fire-and-forget est ajoute a la fin de `closeBatch()` dans `src/actions/batches.ts`
  - Ne bloque jamais la cloture — les erreurs sont loguees en silence
- **Enrichissement `buildBatchAnalysisInput`** dans `src/lib/ai.ts` :
  - race et adresse ajoutees au select Prisma
  - `collectiveBenchmark` injecte dans l'input donne au LLM (mediane mortalite, FCR, marge, prix de vente, stress thermique, scope utilise)
  - Quand assez de donnees existent dans le pool, l'IA compare automatiquement le lot aux lots reels de tout le reseau
- **Cron `app/api/cron/collective-intelligence/route.ts`** :
  - Backfill 30 lots/execution pour les lots anciens sans snapshot
  - Log les stats du pool (total, par type, par region, dernier snapshot)
  - Securise par `CRON_SECRET`, planifie a 3h du matin dans `vercel.json`
- **8 tests unitaires** sur les fonctions pures du benchmark (insight formatting, structure, valeurs limites)
- **Build + lint verts** — aucune regression

### Garanties d'anonymisation

- `BatchOutcomeSnapshot` ne contient aucun `organizationId`, `farmId`, `userId` ou `batchId`
- Les queries de benchmark retournent uniquement des agregats statistiques (percentiles)
- Le pool ne permet pas de retrouver l'identite d'une ferme depuis un snapshot

### Resultat

- Chaque nouvelle cloture de lot enrichit automatiquement le pool collectif
- Des que ~30-50 lots sont dans le pool, le benchmark collectif s'active sur les analyses IA
- L'IA peut desormais dire "votre mortalite est de 5.2% vs mediane collective de 3.5% sur 47 lots similaires" a la place de seuils codes en dur
- Les seuils adaptatifs (`kpi-thresholds.ts`) seront remplaces par des percentiles collectifs en Phase B

### Prochaine session recommandee (Phase B)

- Remplacer les seuils statiques de `kpi-thresholds.ts` par des seuils adaptatifs calcules depuis le pool
- Ajouter une carte UI `BenchmarkCard` sur la page detail du lot pour afficher graphiquement la position du lot dans la distribution collective
- Implementer la detection automatique de patterns (`WEATHER_MORTALITY`, `FCR_BENCHMARK`) via le cron et les persister dans `LearnedPattern`
- Ajouter le formulaire de feedback terrain sur les recommandations IA
